# Bridge AI · Business ↔ Tech Translator

一个基于 Dify Workflow 的角色翻译器：把产品经理、工程师、市场、销售、客户成功之间的表达互相转换，降低跨职能沟通成本。

运营说「我希望页面更快一点」，工程师看到的是具体优化方向；工程师说「支持流式输出」，运营看到的是用户体验价值的描述。

## 快速开始

1. 在 Dify（云版或自部署）中新建空白应用，选择「导入配置文件」（Import DSL）
2. 上传本仓库的 `bridge-ai.yml`
3. 在 设置 → 模型供应商 中配置任一对话模型（项目使用 DeepSeek 对话模型，温度 0.3）
4. 运行：选择说话人角色、目标角色，输入消息即可

## 架构

开始节点（输入契约）→ LLM 节点（翻译）→ 模板转换节点（包装）→ 结束节点（输出契约）

- **开始节点**：三个输入变量 —— source_role / target_role（下拉枚举，受控词表保证下游确定性）、message（段落文本）
- **LLM 节点**：九条行为规则的系统提示词，角色画像式翻译（而非规则枚举），温度 0.3
- **模板转换节点**：固定报告包装（确定性工作用确定性工具，零成本零幻觉）
- **结束节点**：输出契约，交付完整翻译报告

## 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 应用类型 | Workflow | 任务路径确定，需要稳定可控；Agent 的自主性反而带来不可控 |
| 角色输入 | 下拉枚举 | 受控词表，错误预防优于错误处理 |
| 提示词策略 | 角色画像 | 复杂度从 O(n²) 降到 O(n)，加角色只需加一段画像 |
| 模型选型 | 对话模型 | 翻译不需要深度推理；推理模型会泄漏思考过程、慢且贵 |
| 格式化 | 模板节点 | 语义层给 AI，展示层给确定性工具 |
| 防编造 | 显式规则 | 不许编数字、不许编方案、不许替工程师做选型 |

完整决策记录见 `docs/design-decisions.md`，测试用例见 `tests/`。

## 测试用例库

1. 短用例：产品经理 → 工程师（模糊性能诉求）
2. 反向用例：工程师 → 市场（技术概念转体验价值）
3. 边界用例：同角色 / 极短消息 / 中英混合
4. 压力用例：五信息点长文（保真度、数字准确、动因保留、不指定框架）
