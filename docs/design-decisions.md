# Business ↔ Tech Translator · 设计日志

> 记录每一次设计决策与背后的理由。最终用途：GitHub 仓库 README + 面试讲解素材。

## 项目一句话

用 Dify Workflow 做的角色翻译器：把产品、运营、市场、工程师之间的表达互相转换，降低跨职能沟通成本。

## 架构

开始节点（输入契约）→ LLM 节点（翻译）→ 模板转换节点（包装）→ 结束节点（输出契约）

## 关键决策记录

### 1. 为什么用 Workflow 而不是 Chatflow / Agent
- 任务是一次性转换，不需要多轮对话 → 排除 Chatflow
- 任务路径完全确定（输入→翻译→输出），需要稳定可控，不需要 AI 自主规划 → 排除 Agent
- Agent 适合开放式任务，Workflow 适合生产环境的确定性任务

### 2. 为什么不用 RAG / Tool
- 翻译靠的是模型本身对两种语言体系的理解，无需外挂知识库
- 第一阶段不与外部系统交互，无需 Tool
- 扩展方向：若做「公司内部术语翻译器」，RAG 就合适了

### 3. 输入设计：角色用下拉枚举而非自由文本
- 自由文本会产生「PM/产品经理/pm」等脏输入，下游不可控
- 用受控词表把输入收敛，换取下游确定性：错误预防优于错误处理
- 变量名用英文蛇形命名（机器读），显示名中英双语（人读）

### 4. 提示词设计：角色画像而非规则枚举
- 5×5=25 条翻译规则不可扩展；改为描述每个角色的语言风格画像，AI 自行组合
- 复杂度从 O(n²) 降到 O(n)，加新角色只需加一段画像

### 5. 模型选型：对话模型而非推理模型
- 推理模型会先输出思考过程，污染结果、慢、贵
- 翻译是语言转换任务，不需要深度推理
- 温度设 0.3：翻译要求准确保真，不要创造性

### 6. 调优记录（测试驱动）
| 问题 | 修复 | 规则 |
|------|------|------|
| 编造具体指标（「两秒以内」） | 禁止编造数字，给替代出口（建议由工程团队定义目标） | 规则二 |
| 信息不足时编方案（「尽快上线」→ 编整套计划） | 不许编造，先忠实翻译再列缺失信息 | 规则六 |
| 替工程师做架构决策（指定框架） | 只给方向，选型留给工程团队评估 | 规则七 |
| 同角色输入 | 判定为伪需求；现阶段提示词兜底，路线图用表单校验拦截 | 产品设计决策 |

### 7. 格式化分工：语义层 vs 展示层
- AI 只负责语义内容（翻译正文 + 术语解释）
- 固定包装（报告头、分隔线、落款）交给模板转换节点：确定性工作用确定性工具，零成本零幻觉

### 8. 规则优先级：翻译为主，追问为附
- 「给方向」与「要澄清」两条规则在边界输入上打架
- 解法：保真翻译是主任务，澄清追问降级为附录

## 测试用例库
1. 短用例：产品经理→工程师「页面更快一点」
2. 反向用例：工程师→市场「流式输出」
3. 边界用例：同角色 / 极短消息 / 中英混合
4. 压力用例：五信息点长文（见 长文测试用例.txt）
   - 验收：五点保真、数字准确、动因保留、不指定框架、板块完整

## 踩坑笔记（面试素材）
- 误选推理模型导致思考过程泄漏 → 模型选型是产品决策
- 把代码粘进模板框，输出原样打印 → 模板渲染与代码执行是两种机制
- 提示词补丁式修改失控，规则散落两个框 → 单一事实来源，整框替换
- 评估保真度必须对照真实输入，凭印象打分必翻车
